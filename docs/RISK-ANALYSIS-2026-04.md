## 2. 법률·규정 리스크

### 2-1. 이메일 수집 (Layer 2)

**적용 법률:**
- 개인정보보호법 제15조: 개인정보 수집 시 목적·항목·보유기간 고지 + 동의 필수
- 정보통신망법 제50조: 영리 목적 광고성 메일은 사전 수신 동의 필수
- 위반 시: 3,000만원 이하 과태료 또는 5년 이하 징역

**필수 구현 사항:**

```
이메일 구독 폼에 반드시 포함:

□ [필수] 개인정보 수집·이용 동의
  - 수집 항목: 이메일 주소
  - 수집 목적: 뉴스레터 발송 및 투자 정보 알림
  - 보유 기간: 구독 해지 시까지
  - 동의 거부 시 불이익: 뉴스레터 수신 불가

□ [선택] 마케팅 정보 수신 동의
  - 프로모션, 이벤트, 신규 기능 안내

[구독하기]
```

**현재 상태:** InterestRegistration 컴포넌트에 이미 동일한 패턴 구현 완료 (consentCollection + consentMarketing 체크박스). 이 패턴을 뉴스레터 폼에 재사용하면 됨.

**리스크 판정: 관리 가능.** 기존 패턴 재사용으로 법적 요건 충족 가능. 단, 반드시 동의 체크박스 + 개인정보 처리 방침 링크 포함해야 함.

### 2-2. 브라우저 푸시 알림

**적용 법률:**
- 브라우저 푸시는 이메일과 달리 브라우저 자체의 "알림 허용" 네이티브 프롬프트가 동의 역할을 함
- 별도 개인정보 수집 동의가 법적으로 필요하지 않을 수 있으나, 안전하게 수집 목적 안내 권장
- 광고성 푸시는 정보통신망법 적용 → 수신 동의 필요

**리스크 판정: 낮음.** 브라우저 네이티브 프롬프트가 사실상 동의 역할. 단, 광고성 내용이면 추가 동의 필요.

### 2-3. 카카오 OAuth (회원가입)

**현재 상태:** 카카오 OAuth로 수집하는 정보:
- 카카오 계정 고유 ID
- 프로필 사진 (선택)
- 닉네임 (선택)

**이미 개인정보 처리 방침 페이지 존재** (/privacy). 카카오 로그인 시 카카오에서 자체 동의 프로세스 처리.

**리스크 판정: 현재 수준 유지. 추가 리스크 없음.**

### 2-4. 전자상거래법

현재 무료 서비스이므로 전자상거래법 직접 적용 안 됨. 향후 프리미엄 유료 전환 시 검토 필요.

## 3. 기술 리스크

### 3-1. ⚠️ Vercel 크론 97개 — 가장 큰 기술 리스크

**현재:** vercel.json에 97개 크론 등록, 일 861회 실행
**Vercel Pro 제한:** 공식 문서상 Pro는 일일 크론 실행 횟수 제한 있음 (정확한 수는 변동)

**리스크:**
- Vercel이 정책 변경하면 크론 일부가 silent fail 가능
- 새 크론 추가 (push-content-alert, newsletter-weekly) 시 악화
- 현재 왜 97개가 작동하는지 불명 — grandfathered 혹은 내부 soft limit

**대응:**
1. 크론 정리: 97개 중 중복/불필요 제거 → 목표 60개 이하
2. 크론 병합: 유사 목적 크론을 하나로 통합 (예: blog-* 크론 36개 → 10개로)
3. 대안: Supabase pg_cron 또는 Upstash QStash로 일부 이관

### 3-2. Supabase DB 용량 (1.8GB / 8GB)

**성장 시나리오:**

| 테이블 | 현재 | Month 6 | Month 12 |
|--------|------|---------|----------|
| blog_posts | 526MB | 600MB | 700MB |
| apt_rent_transactions | 860MB | 1,200MB | 1,500MB |
| page_views | 6MB | 50MB | 150MB |
| email_subscribers (신규) | 0 | 1MB | 3MB |
| push_subscriptions | 0 | 1MB | 2MB |
| **합계** | **1,838MB** | **~2,500MB** | **~3,500MB** |

**판정:** 12개월 내 8GB 미초과. page_views 테이블만 정기 정리(90일 이상 삭제)하면 안전.

### 3-3. 푸시 알림 iOS 제한

**문제:** iOS Safari는 2023년부터 Web Push 지원하지만, PWA로 설치해야만 작동.
- 일반 Safari 브라우저: 푸시 불가
- 홈 화면 추가 (PWA): 푸시 가능
- Android Chrome: 제한 없이 푸시 가능

**한국 모바일 시장:** Android ~70%, iOS ~30%
**영향:** iOS 유저 30% 중 PWA 미설치 대부분 → 푸시 도달 불가
**대응:** 이메일 뉴스레터가 iOS 유저의 대안 채널

### 3-4. BlogReadGate 제거 후 SSR 영향

**리스크:** 거의 없음.
- 현재: SSR에서 전체 콘텐츠 렌더 → 클라이언트에서 60% 잘림
- 변경: SSR 그대로 + 클라이언트에서도 전체 표시
- SEO: SSR이 이미 전체 콘텐츠 제공 중이므로 변화 없음

### 3-5. SmartSectionGate 하이드레이션 불일치

**리스크:** 중간.
- SSR: 전체 콘텐츠 렌더 (봇 SEO 보호)
- 클라이언트: 마운트 후 특정 섹션 블러
- Next.js 하이드레이션 경고 가능

**대응:** `useEffect` + `useState(false)` 패턴 (BlogReadGate와 동일). 첫 렌더는 SSR과 동일 → 마운트 후 블러 적용 → 하이드레이션 불일치 없음.

### 3-6. Two-Step CTA → OAuth 리다이렉트

**리스크:** 낮음.
- "알림 받을래요?" → YES → "카카오 연결" → `/login?redirect=현재페이지`
- 현재 카카오 OAuth 리다이렉트 이미 구현 완료 (auth/callback)
- 가입 후 원래 페이지로 돌아오는 플로우 정상 작동 중

## 4. 운영 리스크

### 4-1. ⚠️ 솔로 개발자 밴드위스

**현재:** Node(노영진) 1인 개발
**Layer 1~7 구현 예상 공수:**

| Layer | 예상 시간 | 복잡도 |
|-------|---------|--------|
| L1 (전환 최적화) | 4시간 | 낮음 |
| L2 (리드 캡처) | 8시간 | 중간 |
| L3 (재방문 엔진) | 12시간 | 높음 (이메일 인프라) |
| L4 (재방문 전환) | 4시간 | 낮음 |
| L5 (SEO) | 지속적 | 낮음 (기존 시스템 활용) |
| L6 (리텐션) | 8시간 | 중간 |
| L7 (바이럴) | 6시간 | 중간 |

**총 약 42시간** — 풀타임 1주 또는 파트타임 2~3주

**리스크:** 다른 기능 개발/유지보수가 지연될 수 있음
**대응:** Layer 1~2를 먼저 (12시간), 나머지는 점진적 추가

### 4-2. 크론 모니터링

**현재:** 97개 크론 + withCronLogging → cron_logs 테이블에 기록
**추가:** push-content-alert, newsletter-weekly → 2개 추가

**리스크:** 크론 실패를 감지하지 못할 수 있음 (특히 silent fail)
**대응:** GOD MODE 어드민 패널에서 이미 모니터링 가능. 일 1회 자동 health check 크론으로 보완.

### 4-3. 이메일 인프라 운영

**새로운 운영 부담:**
- SPF/DKIM DNS 레코드 설정 (1회)
- 이메일 템플릿 관리
- 수신 거부(unsubscribe) 처리
- 반송(bounce) 처리
- 스팸 필터 회피

**대응:** Resend는 이 중 대부분을 자동 처리 (반송, 스팸 점수, unsubscribe 링크 자동 삽입). 초기 부담은 DNS 설정 1회 + 템플릿 1개 작성.

## 5. 비즈니스·전략 리스크

### 5-1. 콘텐츠 게이팅 해제 후 "무료 라이더" 증가

**우려:** BlogReadGate 제거하면 가입 안 하고 콘텐츠만 소비하는 사람 증가
**현실 판단:**

```
현재: 12,042명 방문 → 60% 잘림 → 16명 가입 (0.13%)
변경: 12,042명 방문 → 100% 공개 → 91명 가입 (0.75%)

"무료 라이더" 증가: 12,042 - 91 = 11,951명 (vs 현재 12,026명)
순 차이: 75명 더 가입, 75명 더 읽기

실제로 잃는 것은 없다. 현재도 99.87%가 가입 안 하고 떠나고 있다.
```

**판정: 리스크 아님.** 현재 게이팅이 전환을 유발하지 않고 이탈만 유발하고 있으므로, 해제해도 손해가 없음.

### 5-2. CTA 과다로 브랜드 이미지 훼손

**우려:** 팝업, 배너, CTA가 너무 많으면 "스팸 사이트" 인식

**대응 (이미 설계안에 반영):**
- 첫 방문자: 최대 2개 CTA (SmartSectionGate + Two-Step)
- 재방문자: 최대 3개
- 동시 표시: 1개만 (window.__kdNudgeActive)
- 닫으면 쿨다운 적용

**판정: 관리 가능.** "최대 2개" 원칙을 지키면 스패미하지 않음. 단, 초기에 너무 많이 넣으면 안 됨 → 실제 구현 시 하나씩 추가하며 반응 확인.

### 5-3. 이메일 스팸 분류

**리스크:** 새 도메인(kadeora.app)에서 발송하는 이메일이 Gmail/Naver Mail 스팸으로 분류될 수 있음

**대응:**
1. SPF/DKIM/DMARC DNS 레코드 설정 (필수)
2. Resend 사용 시 도메인 인증 자동 처리
3. 초기에는 소량(일 100통 이하)으로 시작 → 점진적 증가
4. 구독자가 직접 입력한 이메일에만 발송 (purchased list 사용 금지)
5. unsubscribe 링크 필수 포함

**판정: 관리 가능하지만 주의 필요.** 초기 발송량을 낮게 유지하고, 도메인 인증을 완료해야 함.

### 5-4. 경쟁사 대응

**현재 경쟁 환경:**
- 직방, 호갱노노, 네이버 부동산: 이미 확립된 브랜드
- 카더라의 차별점: 주식 + 부동산 통합, 계산기 142종, AI 분석

**리스크:** 대형 플랫폼이 동일 기능을 추가하면 경쟁 불리
**대응:** 니치(niche) 시장에 집중 — "청약 가점 분석", "실거래가 데이터 블로그" 등 세분화된 키워드에서 우위 확보

## 6. SEO 리스크

### 6-1. BlogReadGate 제거 시 Google 신호 변화

**이론적 우려:** 콘텐츠 게이팅이 "체류 시간"을 인위적으로 늘리고 있었다면, 제거 시 체류 시간 감소 → 순위 하락

**현실 판단:**
```
현재 상황:
- 60% 잘림 → 사용자가 "아 잘렸네" → 뒤로가기 (바운스)
- Google은 빠른 바운스를 부정적으로 해석
- 잘림이 체류 시간을 늘리는 게 아니라 바운스를 증가시킴

변경 후:
- 100% 공개 → 글 끝까지 읽는 사람 증가 → 체류 시간 증가
- 관련글 클릭 → 내부 페이지 이동 → 바운스율 감소
- Google에게 긍정적 신호
```

**판정: SEO에 오히려 긍정적.** 콘텐츠 공개 + 내부 링크 강화는 Google이 좋아하는 패턴.

### 6-2. 내부 링크 대량 변경 시 순위 변동

**리스크:** 59K 블로그에 계산기 링크를 일괄 삽입하면, Google이 "대량 콘텐츠 변경"으로 인식할 수 있음

**대응:**
1. 한 번에 전부가 아니라 배치 처리 (주 1,000편씩)
2. 콘텐츠 자체는 변경 안 함 — 하단 관련 링크 섹션만 추가
3. IndexNow로 변경된 페이지 즉시 알림

**판정: 리스크 낮음.** 내부 링크 추가는 콘텐츠 수정이 아니므로 Google이 부정적으로 보지 않음.

### 6-3. 현재 미인덱싱 상태의 리스크

**현재:** Google/Naver 검색 결과에 카더라가 거의 노출 안 됨
**의미:** SEO 리스크 자체가 없음 — 잃을 순위가 없으므로.
**기회:** 지금 변경해도 SEO 손해가 0. 인덱싱 시작 전에 최적의 구조를 갖추는 것이 유리.

## 7. 데이터·프라이버시 리스크

### 7-1. page_views 테이블의 visitor_id

**현재:** visitor_id는 브라우저 생성 UUID, PII(개인식별정보) 아님
**리스크:** visitor_id와 이메일을 결합하면 행동 추적 가능 → 개인정보보호법 적용 가능

**대응:**
1. 이메일 구독 시 visitor_id를 연결하지 않음 (별도 테이블)
2. 또는 연결하되 개인정보 처리 방침에 명시
3. 90일 이상 page_views 자동 삭제 크론

### 7-2. 전화번호 (InterestRegistration)

**현재:** AES-256-GCM 암호화 저장 (이미 구현)
**리스크:** 복호화 키(ENCRYPTION_KEY) 관리. 유출 시 전체 전화번호 노출.
**대응:** 현재 수준 유지. 키는 Vercel 환경 변수에만 저장.
